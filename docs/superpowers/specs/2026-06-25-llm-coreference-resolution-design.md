# LLM 指代消解设计文档

## 概述

当前 RAG 问答系统只处理了"两者""三者""这些"等显式指代模式，对于"展开说说""详细讲讲""为什么"等隐式口语指代无法识别，导致追问被当作独立查询处理，检索结果为空则直接拒答。

本设计引入 LLM 驱动的指代消解，在系统无法确认查询意图时，利用大模型理解对话历史，将隐式指代恢复为可独立检索的完整问题。

## 触发策略

并非每个问题都需要 LLM 消解。仅在以下条件同时满足时触发：

1. 当前检索结果为空，或者最高得分结果低于阈值（0.2）
2. 历史对话记录不为空

满足条件时，用消解后的问题重新检索一次。如果第二次检索仍然为空，维持原有拒答行为。

这样保证简单查询走规则（快、省），只有系统"搞不定"时才调用 LLM（慢、贵但准）。

## LLM 消解 Prompt

构造专用 Prompt，与最终回答的 Prompt 分离：

```
你是一个指代消解助手。用户正在和一个知识库问答系统对话。
你的任务是根据对话历史，把用户的最新问题改写成一个可以独立检索的完整问题。

对话历史：
用户：{问题1}
助手：{回答1}
用户：{问题2}
助手：{回答2}

用户最新问题：{当前问题}

请直接输出改写后的问题，不要任何解释：
```

LLM 输出即为消解后的问题，直接用于第二次检索。

关键约束：
- 要求 LLM"直接输出，不加解释"，方便代码解析
- 若 LLM 输出与原始问题相同，视为无需消解，保持原样
- 消解用 LLM 复用现有 `OpenAICompatibleLLM` 实例，不额外创建

## 流程变更

改动集中在 `agent.py` 的 `answer()` 方法。

当前流程 → 目标流程：

```
                    ┌─ 规则消解（不变）
                    ├─ 构建查询（不变）
                    ├─ 检索知识库（不变）
现有流程 → 检索 →    └─ 判断结果是否足够好
                    ├─ 结果为空或得分低 → LLM 消解 → 重新检索
                    └─ 结果足够好 → 直接继续
                    └─ 构建 Prompt → LLM 回答（不变）
```

## 改动范围

| 文件 | 改动 |
|------|------|
| `rag_agent/agent.py` | `answer()` 方法新增 LLM 消解分支；新增 `_disambiguate_with_llm()` 私有方法 |
| 其他文件 | 不动 |

预估新增代码约 40-50 行。

## 错误处理

- LLM 调用失败（网络超时、API 错误）：静默回退，使用原始问题检索
- LLM 返回空或无效内容：视为未消解，使用原始问题检索
- 第二次检索仍为空：维持现有拒答行为

## 测试

- 测试场景 1：隐式指代追问（"对他说的详细展开说说"）→ 消解后能检索到相关片段
- 测试场景 2：无关问题（无需消解）→ 不触发 LLM 调用，维持原有检索
- 测试场景 3：空历史对话 + 隐式指代 → 不触发消解，直接拒答
- 测试场景 4：LLM 调用失败 → 回退到原始问题，不崩溃

测试使用现有 `RecordingLLM` 模拟 LLM 返回消解结果。
